# 百科组织规范

## 1. 目标

本规范用于把整个仓库组织成“个人技术百科”，而不是阶段性学习笔记集合。

目录结构需要满足：

- 分类稳定
- 角色清晰
- 易于扩展
- 易于迁移

---

## 2. 一级目录目标结构

后续知识库以以下一级目录为准：

- `00-总览与索引`
- `01-计算机基础`
- `02-编程语言`
- `03-前端开发`
- `04-服务端开发`
- `05-数据与存储`
- `06-架构与工程`
- `07-基础设施与运维`
- `08-AI与智能开发`
- `09-工具与效率`
- `10-实战案例`
- `11-问题排错`
- `98-知识库`
- `99-系统`

一级目录只承担“百科总类”职责，不承担具体文档类型职责。

---

## 3. 二级到四级目录规则

从二级目录开始，每一层目录都要只承担一种结构角色。

常见角色：

- 学习阶段层，如 `01-基础入门`
- 专题分组层，如 `02-核心机制`
- 技术对象层，如 `React`、`Docker`
- 内容承载层，如 `99-参考与归档`

推荐目录格式：

```text
[两位序号]-[分类名]
```

技术对象目录保留官方技术名：

- `React`
- `Docker`
- `Nginx`
- `Node.js`

### 3.1 目录深度上限

新增目录时，层级从一级目录算起，**硬性上限为 5 层**。存量超深目录按软性目标逐步收敛。

示例（合规）：

```text
03-前端开发/           ← 第 1 层
├── 02-前端框架/       ← 第 2 层
│   ├── 01-React/      ← 第 3 层
│   │   ├── 03-Hooks/  ← 第 4 层
│   │   │   ├── 01-useState详解.md  ← 第 5 层（文件）
```

超过 5 层的处理方式：

- 将深层子目录内容提升到上一级，用文件名前缀区分
- 若某个技术子树确实复杂（如 React Native），通过 MOC 链接组织，而非无限嵌套目录
- 新建目录时如果会导致突破 5 层，必须先评估是否可以扁平化

存量超深目录处理：

- 不立即强制扁平化，避免大面积重构
- 后续每次重构该专题时，顺带收敛深度
- 记录在 `03-重构待办.md` 中追踪

---

## 4. 通用专题树

成熟技术专题尽量收敛到以下骨架：

```text
{技术专题}
├── !MOC-{技术专题}.md
├── README.md
├── 01-概念与原理
├── 02-快速上手
├── 03-核心配置与实战
├── 04-高阶与调优
├── 05-常见问题与排错
├── 06-生态与扩展
└── 99-参考与归档
```

不是所有专题都必须一次性补满，但目录角色要优先统一。

---

## 5. 入口文件角色边界

### 5.1 `!MOC-主题名.md`

承担专题入口、索引与导航职责。

适用范围：

- 一级总类
- 成熟技术专题
- 已形成多层结构的知识树

不承担职责：

- 目录说明
- 草稿堆放
- 正文主体

严格定义：

- `!MOC` 是主题导航文件，不是正文文件。
- `!MOC` 必须服务于“看这个主题先看什么、主题下有哪些子专题、核心文章在哪里”。
- `!MOC` 不应展开替代正文的大段知识讲解。

### 5.2 `README.md`

承担目录说明与阅读约定职责。

适用范围：

- 当前目录结构说明
- 阅读顺序建议
- 目录内资产角色说明

不承担职责：

- 替代完整 MOC
- 代替正文系统性承载知识

严格定义：

- `README.md` 是目录说明文件，不是主题导航文件。
- `README.md` 必须解释“这个目录为什么存在、内部如何分层、维护时应如何放置内容”。
- `README.md` 不应承担完整主题索引职责。

### 5.3 编号正文文件

编号正文文件是正式知识文件，是整个知识库的知识本体。

承担职责：

- 承载一个明确主题的知识内容
- 使用 Frontmatter 与模板
- 通过 `moc`、正文链接与所属目录接入知识网络

统一格式：

- `[两位序号]-[主题名].md`

不承担职责：

- 替代 `!MOC`
- 替代 `README.md`
- 承担历史资料收纳

严格定义：

- 编号正文只讲“一个主题本身”。
- 编号正文不负责解释整个目录怎么用。
- 编号正文不负责承担主题总导航。

### 5.4 目录入口治理规则

同一目录层中，入口文件的建设顺序与职责边界应保持稳定。

治理规则：

1. 目录一旦形成稳定分层，优先补齐 `README.md`，用于解释目录结构与维护方式。
2. 目录一旦形成多主题、多子树或稳定阅读路径，再补齐 `!MOC-主题名.md`，用于承担主题导航。
3. 若目录下只有少量正文且尚未形成稳定索引需求，可以暂时只有 `README.md`，但不应用某篇正文兼任入口。
4. 若目录已经出现多个二级子主题、跨目录阅读顺序或阶段性导航需求，应优先补齐 `!MOC`。
5. 同一层目录中，`README.md` 与 `!MOC` 可以并存，但不得写成内容重复的两个导航页。

判定口径：

- 需要解释“这个目录是什么、内部怎么分层、维护时内容放哪里”，使用 `README.md`
- 需要解释“这个主题先看什么、有哪些子专题、核心文章在哪里”，使用 `!MOC`
- 需要解释“某一个知识点本身”，使用编号正文

常见错误：

- 用 `README.md` 承担完整专题索引
- 用 `!MOC` 代替目录说明
- 用 `导航-快速查找.md`、`目录说明.md` 等自定义入口长期并存
- 用一篇正文文章同时承担正文与入口职责

## 6. 归档与原子知识

### 6.1 `99-参考与归档`

统一收纳历史与临时资料。

统一收纳：

- 历史草稿
- 旧版产物
- 网页剪藏
- 无法并入主干专题的资料
- 退役但仍有参考价值的内容

后续统一收敛对象：

- `归档`
- `!小记`
- `收集箱`
- 历史导航页

严格定义：

- `99-参考与归档` 不是正式知识树的一部分，而是保留区。
- 凡是不再适合留在主干阅读路径中的内容，优先迁入此目录。
- `99-参考与归档` 下的文件不应与正式编号正文混排在同一层承担相同角色。
- `99-参考与归档` 下的内容默认不参与站点发布。

统一收敛规则：

- `归档` -> 统一改为 `99-参考与归档`
- `!小记` -> 统一改为 `99-参考与归档`
- `收集箱` -> 统一改为 `99-参考与归档`
- 历史导航页 -> 迁入 `99-参考与归档`，不再作为正式入口保留

说明：

- 这里的“历史导航页”，指过去用于临时导流、快速查找、旧版目录过渡的页面。
- 正式导航职责只能由 `!MOC` 承担。
- 若某个“导航”目录本质上是正式专题，例如 `03-导航系统`，则不按历史导航页处理。

发布规则：

- 所有 `99-参考与归档` 目录统一视为非公开区域。
- 所有收敛到 `99-参考与归档` 的旧内容，在部署后不应暴露。
- Quartz 配置必须默认忽略 `99-参考与归档`、`归档`、`收集箱`、`!小记` 等目录名。

归档目录使用说明：

- 当文件已失去主干阅读价值，但仍需保留检索价值时，迁入 `99-参考与归档`
- 当旧目录正在迁移、暂不能立即删除时，先整体收敛到 `99-参考与归档`
- 当内容只是网页摘录、零散草稿、历史过渡页时，不要继续放在正式正文层
- 迁入归档目录后，不再要求其承担主题入口职责
- 归档文件若仍有有效知识，可在正式正文中以链接方式引用，但不直接作为主阅读路径

不应迁入归档的内容：

- 仍在持续维护的正式编号正文
- 已成为某专题核心阅读路径的文档
- 尚未完成整理但本质属于成熟专题主干的正文

迁入后处理要求：

- 优先保留原始信息，不强制按正文模板重写
- 若文件名明显失真，可做最小必要命名规范化
- 若存在旧入口引用，应在主树入口中改为指向正式主题，而不是继续导向归档页

### 6.2 `98-知识库`

统一作为**永久概念卡片库与原子知识区**。

适用对象：

- 独立概念卡片：不需要深度展开但有独立参考价值的单一概念
- 补充性知识：为正式专题文章提供背景、对比或延伸的小粒度知识
- 待归类知识：尚未确定长期归属的原子知识点

命名方式：

```text
[YYYYMMDD]-[领域]-[核心概念].md
```

定位与规则：

- `98-知识库` 是知识网络的永久节点，不是临时缓存区。概念卡片可以长期保留。
- 不收纳系统性专题正文（需要多章节深度展开的内容应写入正式专题目录）。
- 当同一领域的原子笔记积累到足以形成专题时，**可选择**迁入正式目录，也可保持为独立卡片通过双向引用融入体系。

### 6.3 双向引用规则

`98-知识库` 中的概念卡片通过双向引用融入知识网络：

- **原子笔记 → 正式文章**：原子笔记末尾始终添加 `## 关联知识` 区块，用标准 Markdown 链接指向相关正式文章
- **正式文章 → 原子笔记**：Obsidian 反向链接面板自动感知，无需手动修改正式文章

示例：

```markdown
## 关联知识

- [LLM 原理](../08-AI与智能开发/01-AI基础理论/01-LLM%20原理.md)
- [Transformer 架构](../08-AI与智能开发/01-AI基础理论/02-Transformer%20架构.md)
```

## 7. 文件职责总规则

整个知识库严格执行以下文件职责边界：

- `!MOC`：主题导航文件
- `README.md`：目录说明文件
- 编号正文：正式知识文件
- `99-参考与归档`：历史与临时资料
- `98-知识库`：永久概念卡片与原子知识

判定原则：

1. 如果文件用于说明“这个主题看什么”，它应是 `!MOC`。
2. 如果文件用于说明“这个目录怎么组织”，它应是 `README.md`。
3. 如果文件用于解释某个明确知识点，它应是编号正文。
4. 如果文件只是保留但不再作为主干阅读路径，它应进入 `99-参考与归档`。
5. 如果文件是独立概念卡片或尚未归类的原子知识，它应进入 `98-知识库`。

执行要求：

- 不允许一个文件同时承担两种核心职责。
- 不允许用正文文件替代 `!MOC` 或 `README.md`。
- 不允许把历史资料继续混放在正式正文层。
- 不允许把系统性专题正文放在 `98-知识库`（独立概念卡片不受此限）。
- 不允许将 `归档`、`!小记`、`收集箱` 继续作为长期目录名保留。
- 不允许将历史导航页继续作为正式导航入口暴露在部署站点中。

---

## 8. 当前执行原则

为避免一次性重构失控，执行时遵循以下原则：

1. 先有迁移方案，再做目录迁移。
2. 每次只重构一个一级主题或一个成熟专题。
3. 先整理结构与入口，再处理正文与 Frontmatter。
4. 旧目录若暂时不能删除，先迁入 `99-参考与归档`。
