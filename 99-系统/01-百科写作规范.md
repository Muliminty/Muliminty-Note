# 百科写作规范

## 1. 目标

本规范用于统一知识库正文文件的写法，确保所有文档具备以下特征：

- 可被稳定渲染
- 可被稳定检索
- 可被稳定迁移
- 可被脚本稳定处理

本规范适用于正式体系文章、原子知识、系统规范文档与案例文档。

### 1.1 写作分层原则

所有正式知识笔记统一遵循“通俗易懂、由浅入深”的写作口径。

目标：

- 新手能够先建立最小理解，不因术语密度过高而失去阅读路径
- 有经验的读者能够继续获得原理、边界、实践与细节层面的增量信息
- 同一篇文章同时具备“可入门”和“可深化”的双层价值

写作要求：

- 开头先解释“这是什么、解决什么问题”，再进入原理、配置、推导或案例
- 先给直观认识，再补正式定义、结构细节与实现机制
- 避免一上来堆术语、缩写、结论和抽象概念
- 需要兼顾新手与高手：新手能读懂主线，高手能看到关键约束、设计权衡与实践细节
- 若正文较长，应通过小节递进形成“入门 -> 理解 -> 应用 -> 深化”的阅读层次

## 2. Markdown 与链接规范

### 2.1 原生 Markdown 优先

统一使用标准 Markdown 语法，不依赖特定编辑器的私有写法。

链接统一使用：

```md
[展示文本](相对路径)
```

目标：

- 保证 Obsidian、Quartz 与其他 Markdown 工具兼容
- 保证后续迁移与静态站点构建稳定

### 2.2 双链表达口径

知识关系优先通过正常链接和 `moc` 字段表达，不用随意生成无归属的孤立链接。

要求：

- 文章应尽量挂靠某个 `!MOC`
- 正式文章应尽量补充前置知识
- 跨专题引用要明确主语义，不要把链接堆成标签云

## 3. Frontmatter 规范

所有正式文章顶部必须包含 YAML Frontmatter。

推荐模板如下：

```yaml
---
title: "文章标题"
date: "YYYY-MM-DD"
lastModified: "YYYY-MM-DD"
status: "草稿"
tags: ["标签1", "标签2"]
moc: "[[!MOC-分类名称]]"
stage: "基础入门|框架进阶|工程化实践|质量保障|高级应用"
prerequisites: ["[[前置知识笔记名]]"]
description: "文章描述"
aliases: ["文章别名1", "文章别名2"]
toc: true
---
```

说明：

- `title`：作为文档唯一识别名
- `date`：首次创建时间
- `lastModified`：最后维护时间
- `status`：文章当前生命周期状态
- `tags`：知识分类标签
- `moc`：所在主题的入口锚点
- `stage`：文章所处学习阶段，与目录层级对齐
- `prerequisites`：前置依赖的知识笔记，用于构建阅读路径
- `description`：摘要说明
- `aliases`：异名检索补充
- `toc`：是否显示大纲

补充建议：

- 若文章需要控制是否发布到站点，可按需添加 `publish: true/false`

### 3.1 生命周期状态字段

正式文章统一补充 `status` 字段，并仅使用以下受控值：

- `草稿`：结构未稳定、内容仍在快速变动
- `成熟`：主体内容已可稳定阅读，适合继续增补
- `常青`：长期有效、质量较高、可作为稳定入口引用
- `归档`：保留历史价值，但不再作为主干知识继续扩展

推荐口径：

- 新建正文默认用 `草稿`
- 一轮系统整理后可提升为 `成熟`
- 长期稳定、被频繁引用的主干文章可提升为 `常青`
- 历史方案、旧版实践、迁移来源材料优先标为 `归档`

### 3.2 标签受控规则

标签不再自由生长，统一遵守以下规则：

- 优先使用中文领域标签或官方技术名
- 同一概念只保留一个主标签，不并行创建同义词
- 单篇文章的 `tags` 建议控制在 `3` 到 `5` 个
- 一级领域标签、技术栈标签、内容类型标签可以混合，但不要堆砌

推荐白名单示例：

- 一级领域：`计算机基础`、`编程语言`、`前端开发`、`服务端开发`、`数据与存储`、`架构与工程`、`基础设施与运维`、`AI与智能开发`、`工具与效率`、`实战案例`、`问题排错`
- 技术栈：`JavaScript`、`TypeScript`、`Node.js`、`React`、`Vue`、`Next.js`、`Go`、`Java`、`Docker`、`Nginx`、`Git`、`GitHub`、`Linux`、`Windows`
- 内容类型：`系统规范`、`写作规范`、`目录说明`、`基础入门`、`最佳实践`、`问题排错`

黑名单示例：

- 不再新增并行标签：`front-end`、`FE`、`js`、`ts`、`nodejs`、`reactjs`
- 不使用纯情绪型或过程型标签：`随手记`、`待整理`、`先放这`、`有空再看`

## 4. 文件命名规范

### 4.1 总原则

文件命名遵循三个原则：

- `受控`
- `稳定`
- `可插队`

也就是：

- 不自由发挥命名风格
- 同层长期保持一致
- 后续可以平滑插入新文件

### 4.2 正式体系文章

正式专题文章统一使用：

```text
[两位序号]-[主题名].md
```

示例：

- `01-安装与环境配置.md`
- `02-核心概念.md`
- `03-多服务编排实践.md`
- `04-Compose文件编写指南.md`

约束：

- 文件名只表达当前层主题
- 不重复上级目录已经表达的信息
- 同层统一使用两位数字前缀

### 4.3 原子知识

`98-知识库` 中的原子文件统一使用：

```text
[YYYYMMDD]-[领域]-[核心概念].md
```

示例：

- `20260315-网络-HTTP缓存协商机制.md`
- `20260315-架构-幂等设计模式.md`
- `20260315-效率-代码审查检查清单.md`

### 4.4 特殊文件

- `!MOC-[主题名].md`：主题入口与索引
- `README.md`：目录说明

## 5. 技术名与混排规范

技术名统一采用官方写法：

- `JavaScript`
- `TypeScript`
- `Node.js`
- `Next.js`
- `Nginx`
- `Docker`
- `Claude Code`
- `VSCode`
- `GitHub`

### 5.1 技术名标准写法表

| 领域 | 推荐写法 | 避免写法 |
| --- | --- | --- |
| 语言 | `JavaScript` | `Javascript`、`js` |
| 语言 | `TypeScript` | `Typescript`、`ts` |
| 运行时 | `Node.js` | `NodeJS`、`nodejs` |
| 前端框架 | `Next.js` | `NextJS`、`nextjs` |
| 前端框架 | `React` | `react` |
| 前端框架 | `Vue` | `vue` |
| 样式工具 | `Tailwind CSS` | `TailwindCSS`、`tailwindcss` |
| 服务端组件 | `Nginx` | `nginx`、`Ngnix` |
| 容器工具 | `Docker` | `docker` |
| 容器工具 | `Docker Compose` | `docker compose`、`Docker-Compose` |
| 编辑器 | `VS Code` | `VSCode`、`vscode` |
| 协作平台 | `GitHub` | `github` |
| AI 工具 | `Claude Code` | `claude code` |

使用规则：

- 标题、文件名、目录名优先使用推荐写法
- 若文件名受历史结构限制，至少保证正文标题与说明文字使用标准写法
- 缩写只在上下文已经明确时使用，不作为正式标题主写法

中英文混排时，英文技术词与中文说明之间保留空格。

推荐：

- `React Hooks 原理`
- `Docker Compose 基础`
- `Windows 系统重装与环境初始化清单`

不推荐：

- `ReactHooks原理`
- `Docker-Compose基础`
- `Windows系统重装与环境初始化清单`

### 5.2 文件名与正文分层规则

技术名在文件名/目录名和正文中遵循不同规则：

| 场景 | 规则 | 示例 |
|------|------|------|
| 文件名/目录名 | 允许紧凑形式（省略空格） | `06-TailwindCSS`、`VSCode` |
| 正文标题/说明文字 | 必须使用官方写法 | `Tailwind CSS`、`VS Code` |
| Frontmatter title | 必须使用官方写法 | `title: "Tailwind CSS 配置指南"` |
| 标签 tags | 使用官方写法 | `tags: ["Tailwind CSS"]` |

原因：

- 文件路径含空格可能导致脚本处理和 URL 编码问题
- 正文面向人类阅读，应保持官方写法的可读性

## 6. 受控后缀词表

为减少同义异名，优先从以下后缀中选词：

概念类：

- `基础`
- `原理`
- `机制`
- `架构`

使用类：

- `快速开始`
- `配置指南`
- `使用教程`

实战类：

- `实战`
- `实践`
- `最佳实践`

排错类：

- `问题排查`
- `故障诊断`
- `修复方案`

对比类：

- `对比分析`

速查类：

- `速查`
- `清单`

### 6.1 受控后缀词表总表

| 内容类型 | 推荐后缀 | 适用场景 |
| --- | --- | --- |
| 概念理解类 | `基础` | 入门定义、共识概念、认知建立 |
| 概念理解类 | `原理` | 工作机制、内部逻辑、底层模型 |
| 概念理解类 | `机制` | 协作流程、运行过程、通信方式 |
| 概念理解类 | `架构` | 系统分层、模块关系、全局设计 |
| 使用类 | `快速开始` | 初次上手、最短可运行路径 |
| 使用类 | `配置指南` | 安装、环境、参数、集成配置 |
| 使用类 | `使用教程` | 功能操作、命令实践、流程说明 |
| 实战类 | `实战` | 面向单个案例的完整落地记录 |
| 实战类 | `实践` | 面向一类问题的经验总结与做法 |
| 实战类 | `最佳实践` | 较稳定、可复用、可推广的方法 |
| 排错类 | `问题排查` | 现象定位、排查路径、检查步骤 |
| 排错类 | `故障诊断` | 根因分析、日志判断、问题归因 |
| 排错类 | `修复方案` | 修复动作、回滚策略、验证方法 |
| 对比类 | `对比分析` | 两种以上方案的异同、边界与选择 |
| 速查类 | `速查` | 命令表、参数表、简明要点 |
| 速查类 | `清单` | 步骤检查、执行确认、排查项目 |

选词规则：

- 一个标题只保留一个主后缀，避免“配置指南与最佳实践”这类并列堆叠
- 后缀应表达文档主用途，不要同时兼顾所有内容侧面
- 若内容是长期维护的操作文档，优先用 `配置指南`、`清单`、`最佳实践`
- 若内容主要记录故障处理，不用“踩坑”“记录”，改用 `问题排查`、`修复方案`

尽量避免：

- `总结`
- `整理`
- `踩坑`
- `终极版`
- `大全`
- `手册`

## 7. 图片与资源管理

### 7.1 存储策略

图片与附件资源采用**就近存放**策略：

- 图片存放在与引用文件同目录下的 `img/` 子目录中
- 每个专题的图片独立管理，不跨目录引用

示例结构：

```text
03-前端开发/
├── 02-前端框架/
│   ├── React/
│   │   ├── img/
│   │   │   ├── react-lifecycle.png
│   │   │   └── hooks-flow.png
│   │   ├── 01-核心概念.md
│   │   └── 02-Hooks深入.md
```

### 7.2 命名规范

图片文件名统一使用：

```text
[主题关键词]-[描述].png
```

要求：

- 全小写，单词间用中划线连接
- 不使用空格和中文
- 后缀使用实际格式（`.png`、`.jpg`、`.svg`、`.gif`）

示例：

- `react-lifecycle.png`
- `docker-architecture.svg`
- `flex-layout-demo.gif`

### 7.3 引用方式

统一使用相对路径引用：

```md
![描述文字](./img/文件名.png)
```

禁止：

- 使用绝对路径
- 使用网络图床链接（除非是外部资源的官方图示）
- 在正文中使用 base64 内嵌图片

## 8. 禁止项

以下命名与写法应尽量避免：

- 错别字命名，如 `ngnix`
- 大小写漂移，如 `nginx` 与 `Nginx` 混用
- 情绪化命名，如"保姆级""终极版"
- 目录和文件重复表达同一信息
- 同层同时混用有序与无序命名

## 9. 模板使用原则

正文内容按知识类型选择模板：

- 概念理解类
- 实战应用类
- 问题排错类

模板入口见：

- [模板/00-模板使用指南.md](./模板/00-模板使用指南.md)
- [模板/01-概念理解类模板.md](./模板/01-概念理解类模板.md)
- [模板/02-实战应用类模板.md](./模板/02-实战应用类模板.md)
- [模板/03-问题排错类模板.md](./模板/03-问题排错类模板.md)

补充约束：

- 模板仅用于正式正文文件。
- `!MOC` 与 `README.md` 不套用正文模板。
- `99-参考与归档` 中的历史资料不强制补齐正文模板。
