# 百科写作规范

## 1. 目标

本规范用于统一知识库正文文件的写法，确保所有文档具备以下特征：

- 可被稳定渲染
- 可被稳定检索
- 可被稳定迁移
- 可被脚本稳定处理

本规范适用于正式体系文章、原子知识、系统规范文档与案例文档。

---

## 2. Markdown 与链接规范

### 2.1 原生 Markdown 优先

统一使用标准 Markdown 语法，不依赖特定编辑器的私有写法。

链接统一使用：

```md
[展示文本](相对路径)
```

目标：

- 保证 Obsidian、Quartz 与其他 Markdown 工具兼容
- 保证后续迁移与静态站点构建稳定

### 2.2 双链表达口径

知识关系优先通过正常链接和 `moc` 字段表达，不用随意生成无归属的孤立链接。

要求：

- 文章应尽量挂靠某个 `!MOC`
- 正式文章应尽量补充前置知识
- 跨专题引用要明确主语义，不要把链接堆成标签云

---

## 3. Frontmatter 规范

所有正式文章顶部必须包含 YAML Frontmatter。

推荐模板如下：

```yaml
---
title: "文章标题"
date: "YYYY-MM-DD"
lastModified: "YYYY-MM-DD"
tags: ["标签1", "标签2"]
moc: "[[!MOC-分类名称]]"
description: "文章描述"
publish: true
aliases: ["文章别名1", "文章别名2"]
toc: true
---
```

说明：

- `title`：作为文档唯一识别名
- `date`：首次创建时间
- `lastModified`：最后维护时间
- `tags`：知识分类标签
- `moc`：所在主题的入口锚点
- `description`：摘要说明
- `publish`：是否允许发布
- `aliases`：异名检索补充
- `toc`：是否显示大纲

补充建议：

- 若文章依赖明显前置知识，可增加 `prerequisites`
- 若文章处于草稿期，可将 `publish` 设为 `false`

## 4. 文件命名规范

### 4.1 总原则

文件命名遵循三个原则：

- `受控`
- `稳定`
- `可插队`

也就是：

- 不自由发挥命名风格
- 同层长期保持一致
- 后续可以平滑插入新文件

### 4.2 正式体系文章

正式专题文章统一使用：

```text
[两位序号]-[主题名].md
```

示例：

- `01-安装与环境配置.md`
- `02-核心概念.md`
- `03-多服务编排实践.md`
- `04-Compose文件编写指南.md`

约束：

- 文件名只表达当前层主题
- 不重复上级目录已经表达的信息
- 同层统一使用两位数字前缀

### 4.3 原子知识

`98-知识库` 中的原子文件统一使用：

```text
[YYYYMMDD]-[领域]-[核心概念].md
```

示例：

- `20260315-网络-HTTP缓存协商机制.md`
- `20260315-架构-幂等设计模式.md`
- `20260315-效率-代码审查检查清单.md`

### 4.4 特殊文件

- `!MOC-[主题名].md`：主题入口与索引
- `README.md`：目录说明

## 5. 技术名与混排规范

技术名统一采用官方写法：

- `JavaScript`
- `TypeScript`
- `Node.js`
- `Next.js`
- `Nginx`
- `Docker`
- `Claude Code`
- `VSCode`
- `GitHub`

### 5.1 技术名标准写法表

| 领域 | 推荐写法 | 避免写法 |
| --- | --- | --- |
| 语言 | `JavaScript` | `Javascript`、`js` |
| 语言 | `TypeScript` | `Typescript`、`ts` |
| 运行时 | `Node.js` | `NodeJS`、`nodejs` |
| 前端框架 | `Next.js` | `NextJS`、`nextjs` |
| 前端框架 | `React` | `react` |
| 前端框架 | `Vue` | `vue` |
| 样式工具 | `Tailwind CSS` | `TailwindCSS`、`tailwindcss` |
| 服务端组件 | `Nginx` | `nginx`、`Ngnix` |
| 容器工具 | `Docker` | `docker` |
| 容器工具 | `Docker Compose` | `docker compose`、`Docker-Compose` |
| 编辑器 | `VSCode` | `VS Code`、`vscode` |
| 协作平台 | `GitHub` | `github` |
| AI 工具 | `Claude Code` | `claude code` |

使用规则：

- 标题、文件名、目录名优先使用推荐写法
- 若文件名受历史结构限制，至少保证正文标题与说明文字使用标准写法
- 缩写只在上下文已经明确时使用，不作为正式标题主写法

中英文混排时，英文技术词与中文说明之间保留空格。

推荐：

- `React Hooks 原理`
- `Docker Compose 基础`
- `Windows 系统重装与环境初始化清单`

不推荐：

- `ReactHooks原理`
- `Docker-Compose基础`
- `Windows系统重装与环境初始化清单`

## 6. 受控后缀词表

为减少同义异名，优先从以下后缀中选词：

概念类：

- `基础`
- `原理`
- `机制`
- `架构`

使用类：

- `快速开始`
- `配置指南`
- `使用教程`

实战类：

- `实战`
- `实践`
- `最佳实践`

排错类：

- `问题排查`
- `故障诊断`
- `修复方案`

对比类：

- `对比分析`

速查类：

- `速查`
- `清单`

### 6.1 受控后缀词表总表

| 内容类型 | 推荐后缀 | 适用场景 |
| --- | --- | --- |
| 概念理解类 | `基础` | 入门定义、共识概念、认知建立 |
| 概念理解类 | `原理` | 工作机制、内部逻辑、底层模型 |
| 概念理解类 | `机制` | 协作流程、运行过程、通信方式 |
| 概念理解类 | `架构` | 系统分层、模块关系、全局设计 |
| 使用类 | `快速开始` | 初次上手、最短可运行路径 |
| 使用类 | `配置指南` | 安装、环境、参数、集成配置 |
| 使用类 | `使用教程` | 功能操作、命令实践、流程说明 |
| 实战类 | `实战` | 面向单个案例的完整落地记录 |
| 实战类 | `实践` | 面向一类问题的经验总结与做法 |
| 实战类 | `最佳实践` | 较稳定、可复用、可推广的方法 |
| 排错类 | `问题排查` | 现象定位、排查路径、检查步骤 |
| 排错类 | `故障诊断` | 根因分析、日志判断、问题归因 |
| 排错类 | `修复方案` | 修复动作、回滚策略、验证方法 |
| 对比类 | `对比分析` | 两种以上方案的异同、边界与选择 |
| 速查类 | `速查` | 命令表、参数表、简明要点 |
| 速查类 | `清单` | 步骤检查、执行确认、排查项目 |

选词规则：

- 一个标题只保留一个主后缀，避免“配置指南与最佳实践”这类并列堆叠
- 后缀应表达文档主用途，不要同时兼顾所有内容侧面
- 若内容是长期维护的操作文档，优先用 `配置指南`、`清单`、`最佳实践`
- 若内容主要记录故障处理，不用“踩坑”“记录”，改用 `问题排查`、`修复方案`

尽量避免：

- `总结`
- `整理`
- `踩坑`
- `终极版`
- `大全`
- `手册`

---

## 7. 禁止项

以下命名与写法应尽量避免：

- 错别字命名，如 `ngnix`
- 大小写漂移，如 `nginx` 与 `Nginx` 混用
- 情绪化命名，如“保姆级”“终极版”
- 目录和文件重复表达同一信息
- 同层同时混用有序与无序命名
