# 规范体系重构方案

## 0. 背景与决策记录

2026-05-09 对 `99-系统` 全部规范文件进行了交叉审核（CEO Review），发现五大类问题：

1. **Frontmatter 字段定义分裂** — 写作规范定义了 `status`，模板定义了 `stage`/`prerequisites`，二者互不包含
2. **过时文件未清退** — `03-百科迁移方案.md` 已完成使命、`05-全库规范检查报告.md` 引用已删除的目录
3. **TODO 文件膨胀** — `04-重构TODO.md` 达 927 行，混合战略规划、执行记录和扫描结果三种职责
4. **规范文件自身不合规** — 模板使用指南无 Frontmatter、主文件缺 `status`、`publish: true` 与 Quartz 配置矛盾
5. **缺失模块** — 图片资源存储策略和 Transclusion 使用规则完全空白

### 决策摘要（D1-D7）

| 编号 | 决策 | 选择 |
|------|------|------|
| D1 | 重构力度 | B）选择性重构 — 保留骨架，定点修补 |
| D2 | Frontmatter 字段 | A）合并 — status + stage + prerequisites 三者共存 |
| D3 | 过时文件 | A）归档 + 精简 — 迁入参考与归档，主目录只留活跃文件 |
| D4 | 技术名写法 | A）分层规则 — 文件名允许紧凑形式，正文必须用官方写法 |
| D5 | 缺失模块 | A）图片规范现在写，Transclusion 留 TODO |
| D6 | publish 字段 | B）移除 — 从全库推荐模板和 99-系统 文件中移除 publish，改为按需添加 |
| D7 | 目标结构 | 确认 — 主目录 4 个文件 + 模板 + 归档 |

### 阶段三：补写图片资源规范（D5）

**目标**：在 `01-百科写作规范.md` 中新增图片与资源管理章节。

在 §7（禁止项）之前插入新章节，建议编号为 §7，原 §7 和 §8 顺延为 §8 和 §9：

```markdown
## 7. 图片与资源管理

### 7.1 存储策略

图片与附件资源采用**就近存放**策略：

- 图片存放在与引用文件同目录下的 `img/` 子目录中
- 每个专题的图片独立管理，不跨目录引用

示例结构：

​```text
03-前端开发/
├── 02-前端框架/
│   ├── React/
│   │   ├── img/
│   │   │   ├── react-lifecycle.png
│   │   │   └── hooks-flow.png
│   │   ├── 01-核心概念.md
│   │   └── 02-Hooks深入.md
​```

### 7.2 命名规范

图片文件名统一使用：

​```text
[主题关键词]-[描述].png
​```

要求：

- 全小写，单词间用中划线连接
- 不使用空格和中文
- 后缀使用实际格式（`.png`、`.jpg`、`.svg`、`.gif`）

示例：

- `react-lifecycle.png`
- `docker-architecture.svg`
- `flex-layout-demo.gif`

### 7.3 引用方式

统一使用相对路径引用：

​```md
![描述文字](./img/文件名.png)
​```

禁止：

- 使用绝对路径
- 使用网络图床链接（除非是外部资源的官方图示）
- 在正文中使用 base64 内嵌图片
```

### 阶段四：技术名分层规则（D4）

**目标**：在 `01-百科写作规范.md` §5 中明确文件名与正文的分层写法规则。

```markdown
### 5.2 文件名与正文分层规则

技术名在文件名/目录名和正文中遵循不同规则：

| 场景 | 规则 | 示例 |
|------|------|------|
| 文件名/目录名 | 允许紧凑形式（省略空格） | `06-TailwindCSS`、`VSCode` |
| 正文标题/说明文字 | 必须使用官方写法 | `Tailwind CSS`、`VS Code` |
| Frontmatter title | 必须使用官方写法 | `title: "Tailwind CSS 配置指南"` |
| 标签 tags | 使用官方写法 | `tags: ["Tailwind CSS"]` |

原因：

- 文件路径含空格可能导致脚本处理和 URL 编码问题
- 正文面向人类阅读，应保持官方写法的可读性
```

原 §5.2 以下章节编号顺延。

同时需要修正 §5.1 技术名标准写法表中的 `VSCode` 条目。当前表格将 `VS Code` 列为“避免写法”，与分层规则的“正文必须用官方写法”矛盾。应将该行修正为：

| 编辑器 | `VS Code` | `VSCode`、`vscode` |

即正文使用微软官方写法 `VS Code`，`VSCode` 降为仅限文件名的紧凑形式。
