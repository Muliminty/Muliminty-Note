# 命名规范

## 1. 设计目标

本规范用于统一知识库中的目录命名、文件命名与专题入口命名，核心目标如下：

- 保证新增内容可以快速归位，而不是依赖临时判断。
- 保证同一类知识在命名上具备稳定的可预测性。
- 保证目录扩展后仍然能够维持清晰排序与统一检索。
- 保证后续批量重构、脚本治理、双链维护的成本可控。
- 保证命名体系具备类似图书馆编目系统的可扩展性、可定位性与可维护性。

---

## 1.1 图书馆式编目思路

如果把整个知识库视为一座技术图书馆，那么每个文件名都不只是“标题”，而应当承担最小化的编目职责。

对应关系如下：

- 一级目录相当于图书馆的“总类”，负责表达知识领域。
- 二级及以下目录相当于“分类架位”，负责表达专题层级与结构位置。
- 文件名前缀相当于“排架号”，负责表达同层顺序与后续扩容能力。
- 文件标题主体相当于“书名主题”，负责表达当前文档的唯一核心语义。
- `!MOC` 相当于“主题目录卡”，负责承担导航与索引职责。
- `98-知识库` 相当于“暂编目区”，用于收纳尚未进入正式体系的原子知识。

基于这一思路，命名时不追求单文件的文学性，而追求三件事：

- 放回原位时不需要犹豫。
- 新增同类文档时不需要推翻旧规则。
- 批量检索、重排、迁移时可以被脚本稳定处理。

---

## 2. 命名总原则

### 2.1 单一语义原则

每一个名字只表达一个核心语义，不同时承担“领域 + 类型 + 情绪化描述 + 写作背景”等多重职责。

- 推荐：`03-多服务编排实践.md`
- 不推荐：`Docker 学习总结与踩坑整理终极版.md`

### 2.2 层级分责原则

目录名负责表达“分类位置”，文件名负责表达“具体主题”。

- 目录已表达 `Docker/05-Docker-Compose/`
- 文件只需要表达 `01-Compose基础.md`

避免在目录名和文件名中重复叙述同一信息。

### 2.3 同层一致原则

同一目录层级下，只允许一种命名风格长期存在，不允许混用。

- 要么统一 `01-`、`02-`、`03-` 数字前缀
- 要么统一自然语言标题
- 不允许一部分文件带序号，另一部分完全无序

### 2.4 官方写法优先原则

技术名、产品名、框架名统一采用官方或行业通用写法，不使用个人缩写、误拼写、大小写漂移写法。

- `Nginx`，不写 `nginx` 或 `ngnix`
- `Windows`，不写 `window`
- `Node.js`、`Next.js`、`APNG.js` 使用官方形式
- `Claude Code`、`VSCode`、`GitHub` 保持固定写法

### 2.5 可扩展优先原则

命名规则必须服务于长期扩展，而不是只服务于当前单篇文档的美观。

- 允许未来继续新增 `04-`、`05-`、`06-`
- 避免使用语义模糊的“最终版”“汇总版”“临时整理”

### 2.6 排架稳定原则

同层文件名的前缀需要像图书馆排架号一样稳定，保证未来插入新内容时仍能维持顺序。

- 推荐为体系型文章统一保留两位数字前缀。
- 同一目录下的排序维度只能有一个，不混用“主题优先”和“时间优先”。
- 若预计专题会持续扩张，应优先保留 `01` 到 `99` 的顺序空间。

### 2.7 编目最小重复原则

像图书馆分类号不会把整本书内容重复写一遍一样，文件名也不应重复目录已经表达的信息。

例如：

- 目录：`03-服务端/04-部署与运维/容器化/Docker/01-基础入门/`
- 推荐：`01-安装与环境配置.md`
- 不推荐：`01-Docker安装与环境配置.md`

### 2.8 受控词表原则

为保证全库检索稳定，同类主题应尽量从固定词表中选词，而不是为相近内容创造不同表达。

例如：

- 概念类统一优先使用 `基础`、`原理`、`机制`、`架构`
- 实战类统一优先使用 `实践`、`实战`、`最佳实践`
- 排错类统一优先使用 `问题排查`、`故障诊断`、`修复方案`

这样做的目标，不是限制表达，而是降低“同义异名”造成的知识碎片化。

## 3. 目录命名规则

## 3.1 一级目录

一级目录只表达“知识领域”或“系统角色”，不表达文档类型。

推荐格式：

```text
[两位序号]-[领域名]
```

示例：

- `00-前端体系`
- `01-计算机基础`
- `03-服务端`
- `05-AI辅助开发`
- `99-系统`

约束：

- 使用统一数字前缀，便于根目录排序稳定。
- 目录名保持简洁，不叠加“教程”“合集”“资料库”等泛化词。

## 3.2 二级及以下目录

专题目录优先表达“结构角色”，而不是写作语气。

推荐格式：

```text
[两位序号]-[分类名]
```

示例：

- `01-基础入门`
- `02-核心概念`
- `03-实战应用`
- `04-高阶与调优`
- `05-常见问题与排错`
- `06-生态与扩展`
- `99-参考与归档`

不推荐：

- `小记`
- `随手记`
- `临时整理`
- `各种问题`
- `收集箱`

说明：

- 如果确实需要承载非正式内容，统一收敛到 `99-参考与归档`。
- 如果内容极度零散且尚未确定归属，优先进入 `98-知识库`。

## 4. 文件命名规则

## 4.1 体系型文件

适用于主知识体系中的正式文章。

推荐格式：

```text
[两位序号]-[主题名].md
```

示例：

- `01-安装与环境配置.md`
- `02-核心概念.md`
- `03-多服务编排实践.md`
- `04-Compose文件编写指南.md`

约束：

- 同层文章统一使用两位数字前缀。
- 文件名只表达当前层的主题，不重复上级目录已表达的信息。

例如：

- 目录为 `Docker/05-Docker-Compose/`
- 文件写 `01-Compose基础.md`
- 不必写 `01-Docker Compose 基础.md`

## 4.2 非体系型文件

适用于 `00-经验技巧`、`06-BLOG` 等非标准章节树中的独立文档。

推荐格式：

```text
[主题名].md
```

示例：

- `Cursor 使用技巧.md`
- `MemoBox 部署方案与实践.md`
- `Windows 系统重装与环境初始化清单.md`

约束：

- 不强制加数字前缀。
- 标题直接表达核心主题与场景。

## 4.3 原子知识文件

适用于 `98-知识库` 中的碎片化、单概念、高独立性笔记。

推荐格式：

```text
[YYYYMMDD]-[领域]-[核心概念].md
```

示例：

- `20260313-网络-HTTPS握手流程.md`
- `20260313-架构-缓存击穿与缓存雪崩.md`
- `20260313-效率-代码审查检查清单.md`

约束：

- 日期必须使用 `YYYYMMDD`
- 领域必须简短稳定
- 核心概念必须具体，不能使用“整理”“记录”“总结”

## 5. MOC 与说明文件命名规则

## 5.1 MOC 文件

MOC 文件统一用于承担专题入口和导航索引职责。

推荐格式：

```text
!MOC-[主题名].md
```

示例：

- `!MOC-前端体系.md`
- `!MOC-Docker.md`
- `!MOC-AI辅助开发.md`

约束：

- 一个专题只保留一个主 MOC 文件。
- `!MOC-` 前缀保持固定，不另起别名风格。

## 5.2 README 文件

README 仅用于说明目录用途、维护约定和阅读方式，不承担完整专题索引职责。

适用场景：

- 目录结构说明
- 使用约定说明
- 与 MOC 的角色区分说明

约束：

- 复杂专题可同时存在 `README.md` 与 `!MOC-主题名.md`
- 小目录不建议同时维护两份高度重复的入口文件

---

## 6. 技术名与混排规则

## 6.1 技术名标准化

全库需要固定一套技术名写法，避免同义文件不断增殖。

推荐写法示例：

- `JavaScript`
- `TypeScript`
- `React`
- `Vue`
- `Node.js`
- `Next.js`
- `Nginx`
- `Docker`
- `Docker Compose`
- `Windows`
- `WSL`
- `GitHub`
- `VSCode`
- `Claude Code`

## 6.2 中英文混排规则

中英文混排时，英文技术词与中文描述之间保留空格。

推荐：

- `React Hooks 原理`
- `Docker Compose 基础`
- `Windows 系统重装与环境初始化清单`

不推荐：

- `ReactHooks原理`
- `Docker-Compose基础`
- `Windows系统重装与环境初始化清单`

说明：

- 缩写词与专有名词保持原样，不随意改写。
- 不因为追求“无空格文件名”而牺牲可读性。

---

## 7. 受控后缀词表

为避免同类文章出现大量同义但不统一的标题，建议文件名后缀尽量从以下词表中选择。

### 7.1 概念类

- `简介`
- `基础`
- `原理`
- `机制`
- `架构`

### 7.2 使用类

- `快速开始`
- `配置指南`
- `使用教程`

### 7.3 实战类

- `实战`
- `实践`
- `最佳实践`

### 7.4 排错类

- `问题排查`
- `故障诊断`
- `修复方案`

### 7.5 对比类

- `对比分析`

### 7.6 速查类

- `速查`
- `清单`

不推荐高频使用的模糊后缀：

- `总结`
- `记录`
- `整理`
- `踩坑`
- `笔记`
- `手册`
- `大全`
- `终极版`

---

## 8. 禁止项

以下命名方式应尽量禁止：

- 错别字命名，例如 `ngnix`、`window`
- 大小写漂移，例如 `nginx` 与 `Nginx` 混用
- 情绪化命名，例如 `保姆级教程`、`终极整理`
- 语义空泛命名，例如 `一些总结`、`杂项记录`
- 目录和文件重复表达同一信息
- 在同一层目录中混用有序与无序两种命名风格

---

## 9. 防熵增维护规则

为了防止知识库扩展过程中持续失控，新增内容时应执行以下检查：

1. 先判断内容属于“体系型”还是“原子型”。
2. 先确定目录位置，再命名文件，不允许先写一篇再临时找地方塞入。
3. 先复用已有技术名写法，不重新创造同义词。
4. 先从受控后缀词表中选词，不自由发挥标题风格。
5. 无法归类时，暂存至 `98-知识库`，而不是在专题目录下继续生成新的“临时区”。

---

## 10. 推荐落地口径

如果只保留最核心的一套规则，建议采用以下最小闭环：

- 目录：`[序号]-[分类名]`
- 正式文章：`[序号]-[主题名].md`
- MOC：`!MOC-[主题名].md`
- 原子知识：`[YYYYMMDD]-[领域]-[核心概念].md`
- 归档目录：统一 `99-参考与归档`
- 技术名：统一官方写法
- 混排：中英文之间保留空格
- 临时内容：统一进入 `98-知识库`

---

## 10.1 图书馆式整理落地方案

如果面向整个项目执行命名治理，建议采用“先定编目法，再批量排架”的节奏，而不是直接全库重命名。

### 第一阶段：建立统一编目法

- 固定一级目录作为总类，不再改动根层骨架。
- 固定二级到四级目录的命名模式为 `[序号]-[分类名]`。
- 固定正式文章命名模式为 `[序号]-[主题名].md`。
- 固定 MOC、README、原子知识三类特殊文件的角色边界。

### 第二阶段：建立异常文件清单

优先清点以下文件：

- 含空格但又属于体系型专题的文件。
- 含括号、全角符号、情绪化修饰词的文件。
- 同层中部分带序号、部分不带序号的文件。
- `README.md` 与 `!MOC-主题名.md` 职责重复的目录。
- 技术名大小写不一致或中英文混排不一致的文件。

### 第三阶段：按专题分批治理

建议按“一级专题 -> 二级专题 -> 子专题”的粒度推进，每次只改一个稳定范围，并同步完成：

- 文件重命名
- 目录入口修订
- 文内链接修订
- TODO 进度回填

### 第四阶段：保留扩展余量

- 正式专题坚持数字排架，保留后续新增空间。
- 原子知识坚持日期排架，保证先收纳、后归档。
- 归档资料统一进入 `99-参考与归档`，避免污染主体系。

这一方案的核心不是“一次性改完”，而是让知识库后续每扩展一次，都还能保持像图书馆一样持续整齐。
