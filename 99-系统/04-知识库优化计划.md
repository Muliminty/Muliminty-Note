# 知识库优化计划

## 背景

2026-05-09 对整个知识库管理方案进行了四维审核（CEO 战略 / Design 设计 / Engineering 架构 / DX 体验），综合评分 6.25/10。核心发现：

- 规范设计水平高，但执行落地与日常使用之间存在断层
- `98-知识库` 原子区几乎空置（仅 1 篇），知识入口堵塞
- 对话学习产出缺乏沉淀通路
- Frontmatter 合规率仅约 72%
- 目录最深 11 层，部分模板 YAML 有兼容性风险

本计划按优先级从高到低排列 6 个执行步骤。每步包含：改什么、为什么改、怎么改。

> **修订说明**：2026-05-09 v2 经四维交叉审核后修订。原步骤 7（重写 MOC）已合并到步骤 1，依赖图已修正，新增量化 KPI 和冒烟测试。
>
> **执行记录**：2026-05-09 全部 6 个步骤执行完成。Frontmatter 合规基线：702 个正文文件中 11 个合规（1.6%），主要差距为存量文件缺少 `status` 字段。

## 步骤 1：激活 `98-知识库` — 降低写入门槛并重写 MOC

### 为什么改

`98-知识库` 设计为"先收纳后归档"的原子知识缓冲区，但实际只有 1 篇文件。根因是写入门槛过高——当前规范要求 10+ 个 frontmatter 字段全填，导致日常碎片知识没有低成本的落盘渠道。

### 改什么

1. 在 `01-百科写作规范.md` 的 §3 Frontmatter 规范中，新增 `### 3.3 原子知识精简模式` 子节（接在现有 §3.2 标签受控规则之后）
2. 明确 `98-知识库` 中的文件只要求 **3 个必填字段**：`title`、`date`、`tags`
3. 其余字段（`status`、`moc`、`stage`、`prerequisites`、`description`、`aliases`、`toc`）全部标记为可选
4. 重写 `!MOC-知识库.md`：
   - 删除 OrbitOS 双库分工声明（该仓库在当前 workspace 不存在，引用无法验证）
   - 修正非标准 frontmatter（当前使用 `type: moc` / `created:` / 英文标签，需改为标准 `title` / `date` / 中文标签格式）
   - 内容聚焦于：知识库定位（低门槛原子知识缓冲区）、写入规范（精简 3 字段 + 命名规则）、升级路径（积累 3-5 篇同领域笔记后迁入正式专题）

### 精简模式 frontmatter 模板

```yaml
---
title: "核心概念名"
date: "YYYY-MM-DD"
tags: ["领域标签"]
---
```

### 执行文件清单

- `99-系统/01-百科写作规范.md` — §3 新增 3.3 子节
- `98-知识库/!MOC-知识库.md` — 重写（含 frontmatter 修正）

### 冒烟测试

步骤 1 完成后，立即写入一篇示例原子笔记验证流程：

文件路径：`98-知识库/20260509-示例-精简模式验证.md`，使用精简 3 字段模式（`title` / `date` / `tags`）。确认可正常提交后删除该文件。

## 步骤 2：建立"对话知识沉淀"协议

### 为什么改

用户高频与大模型对话学习，但 `AGENTS.md` 和 `CLAUDE.md` 中没有任何关于"对话产出如何转为笔记"的规定。AI Agent 不知道：何时触发沉淀、最小必填字段是什么、不确定归属时放哪里。

### 改什么

在 `AGENTS.md` 和 `CLAUDE.md` 中同步新增 **§6 对话知识沉淀协议**，内容包括：

#### 6.1 触发条件

当对话中出现以下信号时，AI 应主动提议沉淀：

- 用户明确说"记下来"、"沉淀"、"写入笔记"
- 对话产出了可复用的代码模式、配置方案
- 对话解决了一个具体技术问题（Bug 修复、排错）
- 对话中用户表达"原来是这样"等理解性结论

#### 6.2 沉淀流程

```
1. 判断知识类型 → 选模板（概念/实战/排错）
2. 判断归属目录：
   - 明确属于某专题 → 写入对应专题目录
   - 不确定归属 → 写入 98-知识库/
3. 生成文件：
   - 98-知识库 使用精简模式（3 字段）
   - 正式专题使用完整 frontmatter
4. 命名规则：
   - 98-知识库：YYYYMMDD-领域-核心概念.md
   - 正式专题：[序号]-[主题名].md
```

#### 6.3 默认行为

- AI 不确定归属时，**必须默认写入 `98-知识库/`**，不得直接创建新的专题目录
- AI 不确定知识类型时，使用概念理解类模板
- 建议一次对话沉淀控制在 3 篇以内；若对话确实产出更多独立知识点，可在用户确认后突破上限

#### 6.4 拒绝沉淀

- 当 AI 提议沉淀而用户拒绝时，AI 不再追问，本次对话内不重复提议同一知识点
- 用户可随时主动要求沉淀，不受之前拒绝影响

### 执行文件清单

- `AGENTS.md` — 新增 §6
- `CLAUDE.md` — 新增 §6（内容与 AGENTS.md 保持完全一致）

## 步骤 4：清理模板 frontmatter 中的 YAML 注释

### 为什么改

三大模板（`01-概念理解类模板.md`、`02-实战应用类模板.md`、`03-问题排错类模板.md`）的 frontmatter 中使用了 YAML 注释行（如 `# 🔴 核心必填`）作为分组标记。虽然 Obsidian 能解析，但部分 Markdown 渲染器和 YAML 解析器可能不兼容，且不符合"frontmatter 保持纯净 YAML"的最佳实践。

### 改什么

1. 删除三大模板 frontmatter 中的所有注释行（`# 🔴 核心必填`、`# 🟡 体系建设`、`# 🟢 拓展与控制`）
2. 将分组说明集中写入 `99-系统/模板/00-模板使用指南.md`，不在每个模板中重复嵌入（避免基于模板生成新文件时残留分组说明）

### 改后的 frontmatter 示例（以概念理解类为例）

改前：

```yaml
---
# 🔴 核心必填
title: "{{date}} 什么是？"
date: "{{date}}"
lastModified: "{{date}}"
tags: ["概念", "理论"]
status: "草稿"

# 🟡 体系建设
moc: "[[!MOC-]]"
stage: "基础入门|框架进阶|工程化实践|质量保障|高级应用"
prerequisites: [""]

# 🟢 拓展与控制
description: "关于 xx 概念的深入解析"
aliases: []
toc: true
---
```

改后：

```yaml
---
title: "{{date}} 什么是？"
date: "{{date}}"
lastModified: "{{date}}"
tags: ["概念", "理论"]
status: "草稿"
moc: "[[!MOC-]]"
stage: "基础入门|框架进阶|工程化实践|质量保障|高级应用"
prerequisites: [""]
description: "关于 xx 概念的深入解析"
aliases: []
toc: true
---
```

在 `00-模板使用指南.md` 中集中维护字段分组说明：

| 分组 | 字段 | 说明 |
|------|------|------|
| 🔴 核心必填 | `title`、`date`、`lastModified`、`tags`、`status` | 新建文件时必须填写 |
| 🟡 体系建设 | `moc`、`stage`、`prerequisites` | 正式专题文章建议补齐 |
| 🟢 拓展与控制 | `description`、`aliases`、`toc` | 可选，按需填写 |

## 步骤 5：写入目录深度上限规范

### 为什么改

当前最大目录深度 11 层（如 `03-前端开发/02-前端框架/01-React/13-React生态/React-Native/04-状态管理/README.md`）。超过 4-5 层后，人类很难凭路径定位文件，增加导航和维护成本。

### 改什么

在 `02-百科组织规范.md` 的 §3（二级到四级目录规则）中补充"深度上限"子节：

```markdown
### 3.1 目录深度上限

新增目录时，层级从一级目录算起，**硬性上限为 5 层**。存量超深目录按软性目标逐步收敛。

示例（合规）：

​```
03-前端开发/           ← 第 1 层
├── 02-前端框架/       ← 第 2 层
│   ├── 01-React/      ← 第 3 层
│   │   ├── 03-Hooks/  ← 第 4 层
│   │   │   ├── 01-useState详解.md  ← 第 5 层（文件）
​```

超过 5 层的处理方式：

- 将深层子目录内容提升到上一级，用文件名前缀区分
- 若某个技术子树确实复杂（如 React Native），通过 MOC 链接组织，而非无限嵌套目录
- 新建目录时如果会导致突破 5 层，必须先评估是否可以扁平化

存量超深目录处理：

- 不立即强制扁平化，避免大面积重构
- 后续每次重构该专题时，顺带收敛深度
- 记录在 `03-重构待办.md` 中追踪
```

### 执行文件清单

- `99-系统/02-百科组织规范.md` — §3 新增 3.1 子节

## 步骤 6：开发 Frontmatter 校验脚本 + Pre-commit Hook

### 为什么改

重构待办中列出约 240 个文件缺失 frontmatter，合规率仅约 72%。这严重削弱了按 `status`/`moc`/`prerequisites` 构建知识图谱的能力。需要从"堵增量"和"清存量"两个方向同时推进。

### 改什么

#### 6.1 开发 `scripts/check-frontmatter.js`

功能：

- 扫描指定目录下所有 `.md` 文件
- 检查是否存在 YAML frontmatter（以 `---` 开头和结尾）
- 对于 `98-知识库/` 下的文件，检查精简模式必填字段：`title`、`date`、`tags`
- 对于其他目录下的正文文件，检查完整模式必填字段：`title`、`date`、`tags`、`status`
- 排除 `99-参考与归档/` 目录
- 排除 `README.md` 和 `!MOC-*` 文件（这些有独立规范）
- 排除含 `kanban-plugin` frontmatter 的文件（Obsidian 插件专用格式）
- 输出缺失清单，格式与 `03-重构待办.md` §4.2 一致

#### 6.2 配置 Git Pre-commit Hook

功能：

- 在 `git commit` 前自动运行
- 只检查本次暂存（staged）的 `.md` 文件
- 如果新增或修改的正文文件缺少 frontmatter，**阻止提交**并提示
- 排除 `99-参考与归档/`、`README.md`、`!MOC-*`
- `98-知识库/` 下的文件只检查精简 3 字段

安装方式：

- 优先使用 `.git/hooks/pre-commit` 裸脚本方案（无新依赖），调用 `node scripts/check-frontmatter.js --staged`
- 在 `scripts/README.md` 中记录 hook 安装方法

#### 6.3 远期规划：`--fix` 模式

本轮仅实现检查功能。后续版本可增加 `--fix` 模式，自动为缺失 frontmatter 的文件插入最小骨架模板，加速存量清理。

### 执行文件清单

- `scripts/check-frontmatter.js` — 新建
- `.git/hooks/pre-commit` — 新建
- `scripts/README.md` — 补充 hook 安装说明
