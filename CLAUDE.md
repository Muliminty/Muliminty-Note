# 🤖 AI 辅助内容生成与全库维护规范

为了保证个人知识库的结构化与高质量，所有 AI 辅助生成的文档与系统维护动作，必须严格遵循以下**产品级规范**：

## 1. 语法与链接约束 (原生强兼容)
- **原生优先**：严格使用纯正的 Markdown 原生语法，保证系统和静态部署的最高兼容性。
- **双链绝对规范**：内部引用必须遵循 Obsidian 规范的“双链语法”（`[[文件名]]`），严禁使用易断裂的相对或绝对路径链接。

## 2. 内容层级与目录治理 (体系化存储)
- **目录对齐**：生成或移动文档时，目录树必须符合现有的体系结构规范。
- **专属原子知识池**：引入“原子化”理念，对于单一、碎片化的概念，需统一收敛至专属的「知识库/原子锚点」文件夹，并打满精确的分类标签。

## 3. 内容创作骨架 (逻辑闭环化)
- **预设模板约束**：新笔记或文件夹创建时，AI 输出内容必须强制基于项目中现有的“文件/文件夹官方模板”。
- **模块化行文思维 (按知识类型切换模板)**：
  “是什么 ➡️ 为什么 ➡️ 怎么做” 是宏观心法，但针对具体的知识面，AI 必须切换到对应的**垂直内容模板**进行输出，绝不可“一招鲜吃遍天”：
  
  - **🎯 3.1 概念理解类 (如：什么是闭包？什么是 Docker？)**
    - **是什么**：一句话大白话定义 + 核心特征提取。
    - **为什么**：它解决了什么历史痛点？没有它之前人们是怎么做的？
    - **怎么看**：架构图/工作流模型，或者与相似概念的对比（如 Docker vs 虚拟机）。
    
  - **🛠️ 3.2 实战应用类 (如：Nginx 配置反向代理，React 实现列表渲染)**
    - **是什么**：当前应用场景的简短说明。
    - **怎么做**：**可直接 CV 运行**的代码片段/配置文件，并配合关键步骤的内联注释。
    - **为什么这么做**：核心代码行的原理剖析，以及涉及的设计模式或最佳实践解释。
    
  - **🐛 3.3 问题排错类 (如：内存溢出 OOM 处理，跨域报错解决)**
    - **现象是什么**：具体的报错日志、复现条件。
    - **为什么会这样**：底层排错链路追踪，深层根因分析。
    - **怎么解决**：短期的临时修复方案 (Workaround) + 长期的根本性修复方案 (Root Cause Fix)。

## 4. 资产命名与剥离 (高纯净度与可扩展保障)

为了确保知识库在跨平台同步（Windows/Mac/Linux）、自动化部署（GitHub Pages）、以及长期演进中不崩溃、可检索，**所有文件和文件夹必须严格遵循以下可扩展的命名规范**：

### 4.1 核心命名字符集 (全平台无损兼容)
- **禁用非法与危险字符**：绝对禁止使用 `\ / : * ? " < > |` (系统保留) 以及 `& % $ # @ !` (URL 转义易出错)。
- **空格替换法则**：一律禁止使用空格。单词或复合词组之间，**强制使用连接符 `-` (中划线)**。例如：`React-Hooks-原理解析.md`。

### 4.2 文件命名架构 (可扩展的分类排序机制)
所有笔记和附件，按以下约定组合名称，确保文件列表天然具备逻辑排序能力：

- **体系化知识笔记**：`[序号]-[核心主体]-[具体内容].md`
  - *示例*：`01-Docker-核心概念与架构图.md`, `03-React-实现高阶组件HOC.md`
  - *说明*：序号（如01, 02）用于强制指定系统排序，使其与学习顺位对齐。中划线分离视距，极度清晰。
- **孤立的原子知识/日常Bug**：`[YYYY-MM-DD]-[技术栈]-[问题/知识点].md`
  - *示例*：`2023-11-20-Java-NPE空指针常见场景排查.md`
  - *说明*：对于没有严格前后顺序的日常踩坑，通过“日期前缀”使其在同目录下按时间流自然排序，方便归档和检索。
- **静态资源/图片附件**：`[所属笔记核心名]-[功能序号/后缀].(png|jpg|webp)`
  - *示例*：`01-Docker-核心概念-架构简图01.png`
  - *说明*：附件必须能通过文件名直接看出所属的主笔记是哪篇，彻底解决未来 Obsidian 附件库成为垃圾场的问题。

### 4.3 外部数据隔离机制
- 将网络“剪藏内容”和未消化的资料整体剥离，迁移至独立的「外部归档区/收件箱」，确保主干知识网络 100% 为深度消化后的个人智慧结晶。
## 5. 系统可持续演进 (机器可读与生命周期维护)
- **建立自动化防腐层**：要求格式排版严谨、易于正则提取，为未来无缝对接持续构建（CI）、坏链检查、过期陈旧知识核查等“可持续维护机制”扫清技术障碍。

## 6. MOC 目录树通用规范
无论具体技术栈为何，必须向下兼容这套通用目录结构，以实现“总-分”知识沉淀：

```text
📁 {技术名称} (例如: Docker / Java / 计算机网络)
├── 📄 !MOC-{技术名称}.md        // 🎯 核心入口：当前技术的全局索引与概览 (必有)
│
├── 📁 01-概念与原理           // 🤔 是什么：核心概念、架构图、底层运行原理、发展历史
├── 📁 02-快速上手             // 🚀 怎么起步：安装部署、Hello World、环境搭建、生命周期
├── 📁 03-核心配置与实战         // 💻 怎么用：常用的指令、关键配置文件解析、API、常用最佳实践
├── 📁 04-高阶与调优           // ⚙️ 进阶应用：性能调优、源码级分析、集群与高可用设计、安全防范
├── 📁 05-常见问题与排错         // 🐛 避坑指南：报错排查、开发过程中的常见Bug收集
├── 📁 06-生态与扩展           // 🧩 工具链：相关的周边生态、插件、第三方库分析
│
└── 📁 99-参考与归档           // 📚 附属资料：无法分类的散碎网页剪藏、官方文档链接汇总、废弃的历史产物
```

## 7. 笔记元数据 (Frontmatter) 标准模板
所有笔记必须在开头注入此模板，以保证机器检索的高效性和知识网络的稳固：

```yaml
---
# 🔴 核心必填 (系统运转基础)
title: "文章标题" # 唯一标识：作为文件名的补充，方便肉眼识别与检索
date: "YYYY-MM-DD" # 知识时效标记：用于判断内容新鲜度，配合 lastModified 实现自动化清理或更新旧数据
lastModified: "YYYY-MM-DD" # 维护追踪：标识最后更新时间，避免阅读过期失效的技术内容
tags: ["标签1", "标签2"] # 知识分类标签：将原有 techStack 合并至此，减少记笔记时的分类决策成本，便于全局检索

# 🟡 体系建设 (强烈建议：强化双链与项目结构)
moc: "[[!MOC-分类名称]]" # 知识网络锚点：指明所在 MOC 层级，防止笔记成为孤立节点，构建完整的双链图谱
stage: "基础入门|框架进阶|工程化实践|质量保障|高级应用" # 学习路径对齐：与本项目 README 的阶段划分严格统一
prerequisites: ["[[前置知识笔记名]]"] # 认知依赖提醒：利用双链指明前置要求，降低未来阅读自身代码或复习的学习门槛

# 🟢 拓展与控制 (按需选填：部署与呈现控制)
description: "文章描述" # 博客列表展示：渲染为文章摘要，便于在项目首页快速归纳信息，提升查阅效率
publish: true # 发布准入开关：阻断不完整的草稿意外对外发布，确保 Github Pages 仅展示高质量与结构完整的文章
aliases: ["文章别名1", "文章别名2"] # 软链接防丢机制：若文章存在异名（如 JS/JavaScript），填此可保证 Obsidian 搜索和文档内部跳转稳定不失效
toc: true # 长文导航：针对篇幅较长文章开启阅读侧边栏结构大纲，避免在深入技术细节时丧失宏观位置感
---
```
