# 知识库 MOC

## 文件夹职责
- `98-知识库/` 负责保存偏工程实践、可复用实现、开发经验与技术落地知识
- 它服务“怎么做”，而不是只服务“这是什么”
- 这里的内容主要用于写代码、做系统设计、排查问题和复用实现经验

## 与 OrbitOS 知识库的分工
- `Muliminty-Note/98-知识库` 优先保存“工程版”知识
- `repository-obsidian/40_知识库` 优先保存“概念版”知识
- 如果一个主题同时有概念理解和工程落地，两边都可以有，但职责必须不同

## 如何区分工程版与概念版
- 工程版回答的是：`开发里怎么用、怎么实现、怎么避坑、怎么落地`
- 概念版回答的是：`它是什么、为什么重要、和别的概念有什么关系`
- 如果一篇笔记更像 SDK 用法、架构实现、代码模式、调试经验，放这里
- 如果一篇笔记更像定义、原理、边界、概念关系图，放 OrbitOS `40_知识库`

## 适合放在这里的内容
- API 调用模式
- SDK 用法与封装方式
- 工程项目结构
- 提示词调试经验
- 检索链路实现
- Agent 工作流实现
- 成本、性能、稳定性优化
- 常见报错与排查方法

## 示例
- `大语言模型`
  - 工程版：模型调用方式、上下文管理、流式输出、token 成本控制、模型选型经验
  - 概念版：大语言模型是什么、能力边界、和 Agent / RAG 的关系
- `RAG`
  - 工程版：切片策略、召回链路、向量库选型、检索优化
  - 概念版：RAG 的定义、价值、和微调的区别
- `Function Calling`
  - 工程版：tool schema 设计、多轮调用控制、错误处理
  - 概念版：为什么它能让模型调用工具、它在 Agent 中的角色

## 判断清单
- 如果你在回答“这怎么做”，优先放这里
- 如果你在回答“这是什么”，优先放 OrbitOS
- 如果内容既讲概念又讲实现，拆成两篇，分别服务“理解”和“落地”

## 备注
- 笔记应优先服务未来复用，而不是一次性记录
- 工程经验要尽量写出适用场景、限制条件和踩坑点
- 如果内容已经高度抽象成稳定概念，可在 OrbitOS 再沉淀一篇概念版知识卡
