## 🧩 翻译要求

### 1. 技术准确性 ⚠️

- ✅ 所有技术名词、API 名称、函数、变量、配置项、命令等保持**英文原样**
- ✅ 不得意译或添加原文中不存在的技术信息
- ✅ 参考官方文档（React、Vue、TypeScript、MDN 等）的表达方式
- ✅ 确保专业性与准确性，避免技术错误

**示例**：
- ✅ 正确：使用 `useState` Hook 来管理状态
- ❌ 错误：使用状态钩子来管理状态

### 2. 格式保持 📝

完整保留 Markdown 结构，包括：

- 标题层级（`#`、`##`、`###`）
- 列表（有序 / 无序）
- 表格
- 代码块（```` ```js ```` / ```` ```bash ```` / ```` ```json ```` 等）
- 链接与引用
- 图片与图表

**重要**：
- 代码、命令行示例、JSON、YAML 内容**保持原样不变**
- 代码注释如需翻译，保持注释格式
- 保留所有原始链接（URL 不变）

### 3. 翻译风格 ✍️

- ✅ 中文表达自然流畅、简洁专业，符合开发者阅读习惯
- ✅ 可适度调整语序以提升可读性
- ✅ 避免逐词直译的生硬句式
- ✅ 保持术语一致性，不混用不同译法
- ✅ 使用技术文档常用的表达方式

**术语对照表**（保持一致性）：

| 英文 | 中文 |
|------|------|
| component | 组件 |
| hook | Hook（保持英文） |
| middleware | 中间件 |
| configuration | 配置 |
| deployment | 部署 |
| performance | 性能 |
| optimization | 优化 |

### 4. 语义优化 🎨

- ✅ 对长句进行合理分句，提升阅读体验
- ✅ 若段落为提示、警告或注意事项，使用 Markdown 块引用格式

**示例**：

```markdown
> ⚠️ **注意**：此功能仅在开发环境中可用。

> 💡 **提示**：建议在生产环境中启用缓存。

> ❌ **警告**：此操作不可逆，请谨慎使用。
```

### 5. 输出规范 📤

- ✅ 输出完整 Markdown 文档
- ✅ 不添加任何解释、总结或多余说明
- ✅ 翻译完成后直接输出最终中文版本
- ✅ 保持文档的完整性和可读性

**分段翻译处理**：

若文档过长，可分段翻译，但需：

1. 第一段翻译完成后保存
2. 继续下一段时不需要询问用户
3. 保持上下文衔接自然
4. 直到完整翻译完成

### 步骤 1：分析文档结构

- 识别文档类型（API 文档、教程、参考文档等）
