## 1. 仓库概述

### 1.1 仓库定位

本仓库是一个全栈开发知识体系学习笔记库，采用**学习路径 + 技能分类**的双重组织方式，既便于循序渐进学习，也便于快速查找参考。

### 1.2 组织方式

- **学习路径**：从基础入门到高级应用，循序渐进
- **技能分类**：按技术领域分类，便于快速查找
- **MOC（Map of Content）**：每个知识体系都有对应的 MOC 文件作为索引
- **双链系统**：支持 Obsidian 双链语法 `[[wikilink]]` 和 Markdown 链接

### 1.3 技术栈

- **编辑器**：Obsidian（本地编辑）
- **静态站点生成器**：Quartz 4.0（GitHub Pages 部署）
- **版本控制**：Git + GitHub
- **链接格式**：支持 Obsidian 双链和标准 Markdown 链接

### 1.4 主要目录

- `00-前端体系/` - 前端开发知识体系
- `00-经验技巧/` - 日常开发经验技巧
- `01-计算机基础/` - 计算机基础知识
- `02-代码管理/` - 代码管理相关知识
- `03-服务端/` - 服务端开发知识体系
- `书籍/` - 学习参考书籍和资源
- `xx-归档/` - 归档的旧版本文件（不参与构建）
- `xx-草稿/` - 草稿和未完成的内容（不参与构建）
- `Prompt/` - 仓库维护相关文档（不参与构建）

## 2. 目录结构规范

### 2.1 目录命名规范

#### 2.1.1 顶层目录

- 使用**数字前缀 + 中划线 + 名称**的格式
- 数字前缀用于排序：`00-`、`01-`、`02-`、`03-` 等
- 相同优先级可使用相同数字前缀（如 `00-前端体系/` 和 `00-经验技巧/`）
- 示例：
  - `00-前端体系/`
  - `00-经验技巧/`
  - `01-计算机基础/`
  - `02-代码管理/`
  - `03-服务端/`

#### 2.1.2 子目录

- 同样使用**数字前缀 + 中划线 + 名称**的格式
- 数字前缀表示学习顺序或重要性
- 示例：
  - `01-基础入门/`
  - `02-框架进阶/`
  - `03-工程化实践/`
  - `04-质量保障/`
  - `05-高级应用/`

#### 2.1.3 特殊目录

- `xx-归档/` - 归档目录，使用 `xx-` 前缀表示不参与正常排序和构建
- `xx-草稿/` - 草稿目录，使用 `xx-` 前缀表示不参与正常排序和构建
- `Prompt/` - 仓库维护文档目录，不参与构建
- `书籍/` - 学习资源目录，不参与构建（可选）

**注意**：这些特殊目录在 Quartz 配置中已被忽略，不会出现在生成的静态网站中。

### 2.2 目录层级

- **推荐层级**：不超过 4 层
- **示例结构**：
  ```
  00-前端体系/
  ├── 01-基础入门/
  │   ├── JavaScript/
  │   │   ├── 01-语言核心/
  │   │   │   └── 底层原理/
  │   │   └── 02-内置对象与API/
  │   └── HTML/
  └── 02-框架进阶/
      ├── React/
      └── Vue/
  ```

### 2.3 目录组织原则

1. **按学习路径组织**：从基础到高级，循序渐进
2. **按技术领域分类**：同一技术的相关内容放在一起
3. **便于扩展**：预留扩展空间，便于后续添加内容
4. **保持一致性**：相同类型的目录使用相同的命名规范
5. **避免过深层级**：超过 4 层时考虑重新组织

## 3. 文件命名规范

### 3.1 MOC 文件

#### 3.1.1 命名格式

- **格式**：`!MOC-分类名.md`
- **前缀**：`!MOC-`（Map of Content）
- **命名规则**：
  - 使用中划线分隔单词
  - 分类名使用中文或英文
  - 首字母大小写保持一致（推荐首字母大写）
- **示例**：
  - `!MOC-前端体系.md`
  - `!MOC-React.md`
  - `!MOC-javascript.md`（小写技术名称）
  - `!MOC-状态管理.md`
  - `!MOC-Node.js.md`（保留技术名称中的点）

#### 3.1.2 特殊说明

- `!` 前缀用于在文件系统中排序，使 MOC 文件排在前面
- MOC 文件是知识体系的索引文件，包含完整的知识结构
- 每个主要知识体系都应该有一个对应的 MOC 文件

### 3.2 普通内容文件

#### 3.2.1 命名格式

- **格式**：使用中划线分隔单词
- **规则**：
  - 使用中划线（`-`）分隔单词，不使用下划线或空格
  - 文件名应简洁清晰，能够准确描述内容
  - 使用中文或英文，保持一致
  - 避免特殊字符（除中划线外）
- **示例**：
  - `变量声明.md`
  - `作用域和闭包.md`
  - `Redux-Saga.md`
  - `Vuex-最佳实践.md`
  - `GitHub-Pages部署指南.md`

#### 3.2.2 命名建议

- **技术文档**：技术名称 + 功能描述
  - `React-Hooks原理与实践.md`
  - `Vue-响应式原理.md`
- **实践文档**：技术名称 + 实践类型
  - `Redux-最佳实践.md`
  - `Vuex-Pinia-最佳实践.md`
- **教程文档**：主题 + 类型
  - `GitHub-Pages部署指南.md`
  - `Quartz定制教程.md`
- **概念文档**：直接使用概念名称
  - `作用域和闭包.md`
  - `事件循环机制.md`

### 3.3 README 文件

#### 3.3.1 命名格式

- **格式**：`README.md`
- **位置**：放在需要说明的目录下
- **用途**：说明目录结构、学习路径、设计原则等
- **注意**：根目录的 `README.md` 会被 Quartz 忽略，不会出现在构建结果中

### 3.4 其他文件

- **归档文件**：保持原文件名，放在 `xx-归档/` 目录下
- **草稿文件**：保持原文件名，放在 `xx-草稿/` 目录下
- **图片文件**：使用描述性名称，放在 `static/` 目录或与文档同目录

## 6. 内容格式规范

### 6.1 标题层级

- **一级标题**：`# 标题`（文件主标题）
- **二级标题**：`## 标题`（主要章节）
- **三级标题**：`### 标题`（子章节）
- **四级标题**：`#### 标题`（更细分的章节）
- **五级标题**：`##### 标题`（极少使用）

**建议**：一个文档最多使用到四级标题，保持结构清晰。

### 6.2 列表格式

#### 6.2.1 无序列表

```markdown
- 项目一
- 项目二
  - 子项目一
  - 子项目二
```

#### 6.2.2 有序列表

```markdown
1. 第一步
2. 第二步
3. 第三步
```

#### 6.2.3 任务列表

```markdown
- [ ] 未完成的任务
- [x] 已完成的任务
- ✅ 检查点一
- ✅ 检查点二
```

### 6.3 引用格式

```markdown
> 这是一段引用文字
> 
> 可以包含多行内容
```

### 6.4 代码块格式

#### 6.4.1 行内代码

```markdown
使用 `code` 表示行内代码
```

#### 6.4.2 代码块

```markdown
```语言
代码内容
```
```

**注意**：指定语言类型以获得语法高亮。

### 6.5 分隔线

```markdown
---
```

**用途**：分隔主要章节，提高可读性。

### 6.6 强调格式

- **粗体**：`**粗体**`
- *斜体*：`*斜体*`
- ~~删除线~~：`~~删除线~~`

### 6.7 表格格式

```markdown
| 列一 | 列二 | 列三 |
|------|------|------|
| 内容一 | 内容二 | 内容三 |
| 内容四 | 内容五 | 内容六 |
```

### 6.8 链接格式

#### 6.8.1 外部链接

```markdown
[链接文本](https://example.com)
```

#### 6.8.2 内部链接（Markdown）

```markdown
[链接文本](./路径/文件.md)
[链接文本](../路径/文件.md)
```

#### 6.8.3 内部链接（Obsidian 双链）

```markdown
[[文件名称]]
[[文件名称|显示文本]]
```

**注意**：Quartz 支持两种链接格式，但推荐使用 Markdown 链接以确保兼容性。

#### 6.8.4 带描述的链接

```markdown
[文档标题](./路径/文档.md) — 文档描述
```

### 6.9 图片格式

```markdown
![图片描述](./路径/图片.png)
![图片描述](https://example.com/image.png)
```

**建议**：图片放在 `static/` 目录或与文档同目录，使用相对路径。

## 7. 链接规范

### 7.1 链接类型

#### 7.1.1 相对路径链接（推荐）

- **同级目录**：`./文件.md`
- **子目录**：`./子目录/文件.md`
- **父目录**：`../文件.md`
- **上级子目录**：`../子目录/文件.md`

#### 7.1.2 绝对路径链接

- 从仓库根目录开始：`/00-前端体系/!MOC-前端体系.md`
- **注意**：Obsidian 和 Quartz 对路径的处理可能不同，推荐使用相对路径

#### 7.1.3 Obsidian 双链

- **格式**：`[[文件名称]]` 或 `[[文件名称|显示文本]]`
- **注意**：Quartz 支持双链，但文件名必须准确匹配

### 7.2 链接格式

#### 7.2.1 标准格式

```markdown
[链接文本](./路径/文件.md)
```

#### 7.2.2 带描述格式

```markdown
[文档标题](./路径/文档.md) — 文档描述
```

#### 7.2.3 引用格式

```markdown
- [文档标题](./路径/文档.md) — 文档描述
  - 子项一
  - 子项二
```

### 7.3 链接检查

- **链接有效性**：确保链接的文件存在
- **路径正确性**：使用相对路径，确保路径正确
- **链接更新**：文件移动或重命名时，更新所有相关链接
- **工具检查**：使用 `npm run check:wikilinks` 检查双链引用

## 8. 标签规范

### 8.1 标签格式

- **格式**：`#标签名`
- **位置**：文件末尾，单独一行
- **多个标签**：使用空格分隔

```markdown
#标签1 #标签2 #标签3
```

### 8.2 标签命名

- **使用中文或英文**：保持一致
- **简洁明了**：能够准确描述内容
- **小写字母**：推荐使用小写字母
- **中划线分隔**：多个单词使用中划线分隔

### 8.3 标签示例

```markdown
#前端知识体系 #学习笔记 #知识库
#服务端 #服务器 #部署 #运维 #nginx #nodejs #java
#状态管理 #redux #vuex #pinia #react #vue #前端框架
```

### 8.4 标签分类

- **技术标签**：`#react`、`#vue`、`#nodejs`、`#java`
- **领域标签**：`#前端`、`#服务端`、`#数据库`
- **类型标签**：`#学习笔记`、`#知识库`、`#最佳实践`
- **主题标签**：`#状态管理`、`#性能优化`、`#安全`

### 8.5 标签使用建议

- **适度使用**：每个文档 3-5 个标签即可
- **保持一致性**：相同主题使用相同标签
- **层次化**：使用通用标签和具体标签结合

### 10.1 添加新文档

#### 10.1.1 步骤

## 12. 常见问题

### 12.1 文件命名问题

**Q: 文件名应该使用中文还是英文？**

A: 两种都可以，但要保持一致。推荐使用中文，更易理解。如果使用英文，要使用中划线分隔单词。

**Q: 文件名太长怎么办？**

A: 尽量保持文件名简洁，能够准确描述内容即可。如果必须使用长文件名，使用中划线分隔单词。

**Q: MOC 文件名中的技术名称大小写如何规范？**

A: 遵循技术名称的官方写法。如 `!MOC-javascript.md`（小写）和 `!MOC-React.md`（首字母大写）。

### 12.2 目录结构问题

**Q: 目录层级太深怎么办？**

A: 推荐不超过 4 层。如果目录层级太深，考虑重新组织目录结构，将相关内容包括在同一层级。

**Q: 如何决定目录的命名？**

A: 使用数字前缀 + 中划线 + 名称的格式，数字前缀表示学习顺序或重要性。

**Q: 相同优先级的目录如何命名？**

A: 可以使用相同的数字前缀，如 `00-前端体系/` 和 `00-经验技巧/`。

### 12.3 链接问题

**Q: 链接失效怎么办？**

A: 定期检查链接有效性，文件移动或重命名时更新所有相关链接。可以使用 `npm run check:wikilinks` 工具自动检查链接。

**Q: 应该使用相对路径还是绝对路径？**

A: 推荐使用相对路径，更灵活，便于迁移。Obsidian 和 Quartz 都支持相对路径。

**Q: Obsidian 双链和 Markdown 链接有什么区别？**

A: 
- Obsidian 双链：`[[文件名称]]`，更简洁，但需要文件名完全匹配
- Markdown 链接：`[文本](./路径/文件.md)`，更明确，路径更清晰
- 推荐：在 MOC 文件中使用 Markdown 链接，在内容文件中可以使用双链

**Q: 跨目录链接如何写？**

A: 使用相对路径，如 `../父目录/文件.md` 或 `../../上级目录/文件.md`。

### 12.4 内容填充问题

**Q: 如何开始填充内容？**

A: 从 MOC 文件中的待整理项开始，选择自己熟悉的内容填充。可以先填充基础内容，再逐步填充进阶内容。

**Q: 内容应该多详细？**

A: 根据内容的复杂程度决定。基础内容可以简要说明，复杂内容需要详细说明。提供示例和实践建议。

**Q: 如何保持内容更新？**

A: 定期回顾和更新内容，特别是技术文档。在文档末尾标注最后更新时间。

### 12.5 构建与部署问题

**Q: 为什么某些文件没有出现在构建结果中？**

A: 检查文件是否在忽略列表中（如 `xx-归档/`、`xx-草稿/`、`Prompt/` 等）。

**Q: 如何预览构建结果？**

A: 使用 `npm run preview` 命令，会在本地启动服务器预览构建结果。

**Q: 链接在 Obsidian 中正常，但在 Quartz 中失效？**

A: 检查链接格式，确保使用相对路径。可以使用 `npm run fix:links` 修复链接格式。
