# 📥 技能：对话知识沉淀 (Capture Knowledge)

## 🎯 技能目标

从当前对话中提取值得长期保留的知识（概念理解、代码实战、问题排错），按知识库规范生成标准笔记文件并写入正确位置。

## 📋 前置规范

按需读取（不要一次性全读，避免浪费上下文窗口）：

- `99-系统/01-百科写作规范.md` — Frontmatter 规范、命名规范、受控后缀词表（每次必读）
- 对应模板文件 — 只读将要使用的那一个（概念/实战/排错）
- `99-系统/02-百科组织规范.md` — 仅在写入正式专题目录时读取

## 🧠 知识提取规则

### 1. 识别可沉淀知识

从对话中提取符合以下条件的内容：

- 对某个概念建立了清晰理解（定义、原理、架构）
- 产出了可复用的代码模式、配置方案、命令组合
- 解决了一个具体技术问题（报错、排查、修复）
- 形成了可复用的决策依据或最佳实践

**不应沉淀的内容**：

- 纯闲聊、项目管理讨论
- 纯执行类对话（文件重构、批量重命名、代码格式化）
- 已有笔记覆盖的重复知识
- 过于碎片化、缺乏独立价值的信息

### 1.1 去重检查

写入前，用文件名或 title 关键词搜索目标目录，检查是否已存在相似笔记。若有疑似重复，向用户提示并询问是补充现有笔记还是新建。

### 2. 判断知识类型

| 类型 | 判断信号 | 对应模板 |
|------|----------|----------|
| 概念理解 | "是什么"、"为什么"、原理、架构、对比 | `01-概念理解类模板.md` |
| 实战应用 | "怎么做"、代码实现、配置方案、环境搭建 | `02-实战应用类模板.md` |
| 问题排错 | 报错日志、排查过程、Bug 修复 | `03-问题排错类模板.md` |

不确定时，默认使用概念理解类模板。

### 3. 判断归属目录

```
判断流程：
1. 知识是否明确属于某个已有专题？
   → 是：写入该专题目录，使用完整 frontmatter（10 字段）
   → 否：进入第 2 步
2. 知识是否是独立概念或原子知识？
   → 是：写入 98-知识库/（永久概念卡片），使用精简 frontmatter（3 字段）
   → 否：写入 98-知识库/，后续可选迁入正式专题
```

### 3.1 序号确定规则

写入正式专题目录时，先列出目标目录下已有文件，取最大序号 +1 作为新文件序号。

### 正文要求

- 使用对应模板的骨架结构
- 内容源自对话，但需要重新组织为结构化笔记，不是对话记录的复制粘贴
- 遵循“通俗易懂、由浅入深”的写作口径
- 中英文混排保留空格，技术名使用官方写法

### 篇幅控制

- `98-知识库` 原子笔记：50-150 行，精简优先
- 正式专题文章：不超过 300 行
- 优先保留结论和代码，省略探索性推导过程（除非推导过程本身是知识点）

### 纠错规则

- 如果对话中发生过修正（AI 给了错误回答后被纠正），只沉淀修正后的最终结论
- 从对话末尾向前扫描，优先捕获最终结论而非中间过程

## 📝 输出规范

### 写入 `98-知识库/` 时

文件名：`YYYYMMDD-领域-核心概念.md`

Frontmatter（精简模式，仅 3 个必填字段）：

```yaml
---
title: "核心概念名"
date: "YYYY-MM-DD"
tags: ["领域标签"]
---
```

### 写入正式专题目录时

文件名：`[两位序号]-[主题名].md`

Frontmatter（完整模式）：

```yaml
---
title: "文章标题"
date: "YYYY-MM-DD"
lastModified: "YYYY-MM-DD"
status: "草稿"
tags: ["标签1", "标签2"]
moc: "[[!MOC-分类名称]]"
stage: "基础入门"
prerequisites: [""]
description: "文章描述"
aliases: []
toc: true
---
```

## 🚀 执行流程

触发方式：

- **被动触发**：用户说“沉淀”、“记下来”、“写入笔记”或类似指令
- **主动提议**：当对话出现收尾信号（“谢谢”、“搞定了”、“没问题了”）且对话中产生了可沉淀知识时，AI 可主动轻量提议一次（“本次对话涉及 X，是否需要沉淀？”），仅限一次，被拒绝后不再提议

### 步骤 1：提取与确认

**标准模式**（多个知识点或用户未指定内容）：

1. 回顾当前对话，提取可沉淀的知识点（通常 1-3 个）
2. 向用户展示提取结果，包括：
   - 知识点标题
   - 知识类型（概念/实战/排错）
   - 目标路径（`98-知识库/` 或具体专题目录）
3. 等待用户确认或调整

**快捷模式**（用户已明确指定内容，如“把刚才 Docker 的知识记下来”）：

1. 跳过清单确认，直接生成并写入
2. 在步骤 4 报告中展示完整结果

**示例输出**：

```
本次对话可沉淀以下知识：

1. 📥 Docker 多阶段构建原理
   - 类型：概念理解
   - 路径：98-知识库/20260509-容器-Docker多阶段构建原理.md

2. 📥 Nginx 反向代理跨域配置
   - 类型：实战应用
   - 路径：07-基础设施与运维/02-Web服务器/Nginx/05-反向代理跨域配置.md

是否确认？可以调整类型、路径或去掉不需要的条目。
```

### 步骤 2：读取规范与模板

用户确认后，静默读取：
- 对应的模板文件
- 目标目录下的 `!MOC` 和 `README.md`（如果写入正式专题）

### 步骤 3：生成并写入

1. 按模板骨架生成笔记内容
2. 写入目标路径
3. 如果写入正式专题，在对应 `!MOC` 的索引末尾追加 `- [[file-name]]` wikilink
4. 如果写入 `98-知识库/`，在笔记末尾添加 `## 关联知识` 区块，链接相关的正式专题文章（使用相对路径的标准 Markdown 链接）

### 步骤 4：报告

输出写入结果：
- 文件路径
- Frontmatter 摘要
- 如有联动更新（MOC 等），一并说明
- 末尾加一句：`如需调整，直接告诉我修改内容或删除文件`

多篇写入策略：默认一次性全部写入，用户要求时才逐篇确认。

## ⚠️ 约束规则

- **一次对话建议沉淀 ≤ 3 篇**；超过时提醒用户后续批量整理，用户确认后可突破上限
- **不确定归属时，必须默认写入 `98-知识库/`**，不得自行创建新的专题目录
- **用户拒绝沉淀时，不再追问**，本次对话内不重复提议同一知识点
- **严禁修改** `quartz.config.ts` 和 `index.md`
- **标签使用中文领域标签**，遵守 `01-百科写作规范.md` §3.2 标签受控规则
- **文件名遵守受控后缀词表**，避免使用"总结"、"整理"、"踩坑"等黑名单词
